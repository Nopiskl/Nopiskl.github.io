- **直通模式**：传感器/前级输入的数据，直接按时序送入 ISP 处理，不经过“先写 DDR 再读回”这条环路。
- **回读模式**：前级先把 raw 帧写到内存（或内部 raw write buffer），再由 `RAWRD/DMARX` 读回来送 ISP 再处理。它常用于 HDR 多帧/多曝光重组、离线或半离线处理、以及某些 VICAP/ISP 协同场景。

## 三、`CIF -> ISP` 是怎么接进来的

在 `rkisp_create_links()` 里，如果上游实体类型是 `MEDIA_ENT_F_PROC_VIDEO_COMPOSER`，驱动把它认作 `CIF` 输入：

```392:420:kernel/drivers/media/platform/rockchip/isp/dev.c
if (type == MEDIA_ENT_F_CAM_SENSOR) {
	dev->isp_inp = INP_DVP;
	ret = media_create_pad_link(&sensor->sd->entity, pad,
		&dev->isp_sdev.sd.entity, RKISP_ISP_PAD_SINK, en);
} else if (type == MEDIA_ENT_F_PROC_VIDEO_COMPOSER) {
	dev->isp_inp = INP_CIF;
	ret = media_create_pad_link(&sensor->sd->entity, pad,
		&dev->isp_sdev.sd.entity, RKISP_ISP_PAD_SINK, en);

	// mipi-phy -> csi -> isp
	dev->isp_inp = INP_CSI;
	// ...
```

这个地方很关键：

- 传感器直连 DVP 时，`INP_DVP`
- MIPI 经过 CSI 子模块时，`INP_CSI`

所以你可以把 `INP_CIF` 理解成：

> ISP 的上游不是原始 sensor subdev，而是 CIF/VICAP 这一层已经整理好的输入。

## 四、直通模式到底是什么

### 1）从驱动视角定义

“直通”不是代码里单独叫 `direct mode` 的一个大枚举，而是相对“readback”而言的概念。

对 ISP 来说，只要输入来自：

- `INP_CSI`
- `INP_DVP`
- `INP_LVDS`
- `INP_CIF`

这基本都属于**在线/直通路径**：上游时序来的帧，直接进 ISP。

比如 pipeline 打开时，如果输入是这些类型，会配置 CSI patch：

```285:311:kernel/drivers/media/platform/rockchip/isp/dev.c
if (dev->isp_inp & (INP_CSI | INP_RAWRD0 | INP_RAWRD1 | INP_RAWRD2 | INP_CIF))
	rkisp_csi_config_patch(dev);
```

也就是说 `INP_CIF` 和 `INP_CSI` 一样，都被视作“在线输入通道”的一类。

### 2）直通的典型数据流

最典型是：

- sensor -> mipi/csi -> isp  
或
- cif/vicap -> isp

然后 ISP 直接往输出路径送：

- MP
- SP
- BP/FBC 等
- stats/params 并行工作

这种模式下，ISP处理的是**当前实时流**，不是从 DDR 回来的历史 raw 帧。

### 3）直通的特点

- 延迟低
- 链路简单
- 适合普通预览/录像
- 参数按帧同步下发
- 不需要额外的 raw read 节点参与主链路

## 五、回读模式是什么

回读模式的关键是 `dmarx.c` 里的 `rawrd` 逻辑。

### 1）驱动里有两类“DMA输入”

在 `dmarx.c` 里能看到两种不同的输入流配置：

- `dmarx_fmts[]`：普通 `DMARX`
- `rawrd_fmts[]`：`RAWRD0/1/2`

而 `rawrd_config_mi()` 配的是 `CSI2RX_RAW_RD_CTRL` 和 `MI_RAWx_RD_BASE`，明显就是“raw read back”：

```303:340:kernel/drivers/media/platform/rockchip/isp/dmarx.c
static int rawrd_config_mi(struct rkisp_stream *stream)
{
	struct rkisp_device *dev = stream->ispdev;
	u32 val;

	val = rkisp_read(dev, CSI2RX_DATA_IDS_1, true);
	val &= ~SW_CSI_ID0(0xff);
	// 按 fourcc 选择 RAW8/10/12 或 YUV422
	// ...
	rkisp_unite_write(dev, CSI2RX_RAW_RD_CTRL,
			  stream->memory << 2, false);
	rkisp_unite_write(dev, CSI2RX_DATA_IDS_1, val, false);
	rkisp_rawrd_set_pic_size(dev, stream->out_fmt.width,
				 stream->out_fmt.height);
	mi_raw_length(stream);
	return 0;
}
```

这段的含义就是：

- 配置 raw read 通路的数据类型
- 配置回读图像尺寸
- 配置 raw read 的内存长度/地址
- 把内存中的 raw 数据作为 ISP 输入再送进去

### 2）buffer 地址怎么更新

在 `update_rawrd()` 里，当前 buffer 队列中的地址会被写进 `MI_RAWx_RD_BASE`：

```342:375:kernel/drivers/media/platform/rockchip/isp/dmarx.c
static void update_rawrd(struct rkisp_stream *stream)
{
	struct rkisp_device *dev = stream->ispdev;
	// ...
	if (stream->curr_buf) {
		val += stream->curr_buf->buff_addr[RKISP_PLANE_Y];
		rkisp_write(dev, stream->config->mi.y_base_ad_init, val, false);
		// unite 模式下写另一半地址
		// ...
	}
```

这就说明：

- `RAWRD` 设备对应的是“用户/驱动提供一块内存”
- ISP 从这块内存中把 raw 帧读回来继续处理

这就是“回读”。

## 六、HDR 场景下为什么特别强调回读

`capture.c` 里有很直接的注释：

```55:76:kernel/drivers/media/platform/rockchip/isp/capture.c
		max_buf = HDR_MAX_DUMMY_BUF;

		max_buf = 0;
}
```

并且下面清楚区分了：

- `HDR_RDBK_FRAME1`
- `HDR_RDBK_FRAME2`
- `HDR_RDBK_FRAME3`

以及它们分别占用哪些 DMA 通道：

```76:98:kernel/drivers/media/platform/rockchip/isp/capture.c
/*
 * normal: q_tx[0] to dma0
 *	   q_tx[1] to dma1
 * rdbk1: using dma2
	   q_tx[0] to dma2
 * rdbk2: using dma0 (as M), dma2 (as S)
 *	   q_tx[0] to dma0
 *	   q_tx[1] to dma2
 * rdbk3: using dma0 (as M), dam1 (as L), dma2 (as S)
 *	   q_tx[0] to dma0
 *	   q_tx[1] to dma1
 *	   q_tx[2] to dma2
 */
```

这说明回读模式特别适合 HDR：

- 多曝光原始帧先写出
- 再按 ISP 要求从不同 raw read 口读回
- 再做 merge / DRC / HDR 处理

所以你可以把 **回读模式** 理解成：

> ISP 不是“直接吃传感器当前这一帧”，而是“吃已经写入内存的 raw 帧”，必要时一帧、两帧、三帧配合，用于 HDR 或复杂处理。

## 七、直通与回读的核心区别

可以简单对比一下：

| 项目 | 直通 | 回读 |
|---|---|---|
| 输入来源 | CSI/DVP/LVDS/CIF 实时流 | DDR/内部 raw write buffer |
| 时序关系 | 在线实时 | 离线或半在线 |
| 延迟 | 低 | 较高 |
| 常见用途 | 普通预览、录像 | HDR 多帧、raw 重处理、联动回灌 |
| 涉及节点 | ISP sink + MP/SP 等 | `RAWRD/DMARX` + ISP |
| buffer 管理 | 输出路径为主 | 输入/输出都要做 DMA 管理 |

## 八、你提到的“ISP 直通/回读”，在 Rockchip 里还常有一个“在线切换”味道

`dmarx_frame_end()` 里有一个很有代表性的逻辑：

```431:452:kernel/drivers/media/platform/rockchip/isp/dmarx.c
if (rx_buf->is_switch && stream->id == RKISP_STREAM_RAWRD2) {
	switch (dev->rd_mode) {
	case HDR_RDBK_FRAME3:
		dev->rd_mode = HDR_LINEX3_DDR;
		break;
	case HDR_RDBK_FRAME2:
		dev->rd_mode = HDR_LINEX2_DDR;
		break;
	default:
		dev->rd_mode = HDR_NORMAL;
	}
	dev->hdr.op_mode = dev->rd_mode;
	rkisp_unite_write(dev, CSI2RX_CTRL0,
			  SW_IBUF_OP_MODE(dev->hdr.op_mode), true);
	// ...
	dev_info(dev->dev, "switch online seq:%d mode:0x%x\n",
		 rx_buf->sequence, dev->rd_mode);
}
```

这里的含义是：

- 某些 HDR readback 帧处理完后
- 驱动会把模式切回更常规的在线模式
- 日志里直接写着 `switch online`

所以 Rockchip 这套驱动中，“回读”并不一定是全流程永远离线，它可以是：

- 某几个帧阶段先走回读
- 然后再切回在线/直通

这就是很多厂商 ISP 驱动里常说的“半在线 HDR”。

这个最直接的线索在 `capture.h` 的命名里：

以及流枚举：

下面按功能解释。

---

### 1）`rkisp_mainpath` / MP

主输出路径。

用途：

- ISP 处理后的主图输出
- 一般给录像、拍照主码流用
- 分辨率能力最大，画质路径最完整

通常你可以理解为“主路 full-resolution output”。

---

### 2）`rkisp_selfpath` / SP

副输出路径。

用途：

- ISP 处理后的次级输出
- 常用于预览、小分辨率码流、算法旁路消费
- 分辨率和缩放能力通常比 MP 弱

一般相当于“预览路 / 小图路”。

---

### 3）`rkisp_rawwr0/1/2/3` / DMATX0~3

这是 **RAW write** 节点，不是普通 YUV 输出。

用途：

- 把 ISP 前级或 HDR 分路得到的 raw 数据写到内存
- 给 HDR 回读、多帧处理、raw dump 使用
- `capture.c` 中 HDR buffer 分配就是围着它们转的

从代码注释看，HDR 不同模式会映射到不同 DMA 通道：

- `rawwr0/1/2` 常用于多曝光 raw 保存
- `rawwr3` 在某些 sensor fmt 配置里也会参与

你可以把它们理解成：

> ISP/raw 接收链上的原始帧写出节点，主要服务于 HDR / 回读 / 调试抓 raw。

---

### 4）`RAWRD0/1/2` 对应的 video 节点

这几个名字在 `dmarx.c` 里属于 `dmarx_dev.stream[]`，虽然你这次没直接问到，但它们是“回读模式”的核心。

用途：

- 从内存读 raw/YUV 回来送进 ISP
- 是 `readback` 的输入口
- 对应 `INP_RAWRD0/1/2`

如果系统走 HDR readback，那么 `rawwrX` 和 `rawrdX` 会形成一组前写后读的关系。

---

### 5）`rkisp_fbcpath`

FBC 输出路径。

用途：

- 输出压缩格式帧，给后级高效传输/存储
- 常用于和 `ispp` 或显示/编解码链路做高带宽优化

这个更偏平台能力相关，不是所有版本都一样强。

---

### 6）`rkisp_bypasspath` / BP

旁路路径。

用途：

- 某些版本上做较少 ISP 处理、或者走特定数据旁路
- 适合特殊场景下低处理延迟或保留某类数据格式

这个名字已经说明它不是典型主图路径，而是“绕过部分常规处理”的路径。

---

### 7）`rkisp_mainpath_4x4sampling` / `rkisp_bypasspath_4x4sampling`

下采样路径。

用途：

- 给算法做低分辨率采样图
- 降低带宽/内存压力
- 常用于分析、预处理、AI 辅助输入等

名字里的 `4x4sampling` 已经很明确：做稀疏采样输出。

---

### 8）`rkisp_lumapath`

亮度路径。

用途：

- 输出亮度/统计相关数据
- 常用于某些 AE/AWB/辅助算法
- 不是普通彩色视频输出节点

---

### 9）`rkisp_iqtool` / `vir`

虚拟/IQ 工具节点。

用途：

- 给调试、校参、IQ tool 使用
- 不是标准视频预览流
- 更像用户态调试/控制辅助通道

---

## 十一、如果从“实际链路”看，常见几种工作方式

### 模式 A：普通在线预览/录像

链路大概是：

- sensor / csi / cif -> isp
- isp -> mainpath/selfpath
- 同时：
  - `statistics` 输出统计
  - `input-params` 输入每帧调参

这是最标准的“直通模式”。

---

### 模式 B：HDR 半在线/回读

链路大概是：

- sensor -> rawwr0/1/2 先写 raw
- rawrd0/1/2 再读回 ISP
- ISP 做 HDR merge / DRC / 后处理
- 再从 MP/SP/BP/FBC 输出

这是你问的“回读模式”最典型形态。

---

### 模式 C：CIF/VICAP 联动 ISP

链路大概是：

- CIF/VICAP 作为前级输入源
- `dev->isp_inp = INP_CIF`
- CIF 输出 -> ISP sink
- ISP 正常工作

这从 ISP 角度仍然更接近“在线/直通输入”，只是上游不再是 sensor，而是 CIF 组合器。

---

## 十二、怎么快速判断当前系统是直通还是回读

你如果在板子上排查，重点看这几类信号：

### 1）`isp_inp` 是什么
- `INP_CSI / INP_CIF / INP_DVP / INP_LVDS`：偏直通
- `INP_RAWRD0/1/2 / INP_DMARX_ISP`：偏回读

### 2）是否启用了 `rawwr/rawrd`
- 有 raw write 和 raw read 节点参与，基本就是 readback

### 3）HDR 模式值
- `HDR_RDBK_FRAME1/2/3`：明显回读
- `HDR_LINEX2_DDR / HDR_LINEX3_DDR / HDR_NORMAL`：更偏在线/普通处理

### 4）procfs / log
`procfs.c` 里有 readback 运行信息打印，像：

```129:129:kernel/drivers/media/platform/rockchip/isp/procfs.c
seq_printf(p, "%-10s mode:frame%d (frame:%d rate:%dms %s time:%dms frameloss:%d) cnt(total:%d X1:%d X2:%d X3:%d)\n",
```

这个就是驱动对 readback 运行状态的统计输出入口。

---

## 十三、把“直通 / 回读”用一句更工程化的话说

你可以这么理解：

- **直通**：ISP 直接处理“当前到达的实时输入流”
- **回读**：ISP 处理“之前写入内存、再由 DMA 读回来的帧”

前者强调实时链路，后者强调 memory-based 重灌处理。

---

## 十四、如果只看你这个 Rockchip 实现，最重要的几个认知点

### 1）`INP_CIF` 不等于回读
它只是说明 ISP 的上游是 CIF/VICAP，不一定走 DDR 回灌。

### 2）真正的回读关键在 `RAWRD/DMARX`
回读模式核心实现不在 `cif/`，而在 `isp/dmarx.c` 这套 raw read 输入。

### 3）`rawwrX` 和 `rawrdX` 往往成对理解
- `rawwrX`：写出 raw
- `rawrdX`：读回 raw

### 4）`stats` / `params` 不是画面节点
它们是 ISP 的 3A 元数据通道。
