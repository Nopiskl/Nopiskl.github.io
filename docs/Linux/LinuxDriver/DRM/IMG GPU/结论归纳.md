 DRM/KMS 当前路线

DRM/KMS 路线的核心链路是：

PVR userspace
  -> PVR bridge / pvrsrvkm
  -> RGX GPU 渲染到 buffer
  -> 导出 dma-buf
  -> 用户态 KMS client / compositor
  -> sunxi-drm
  -> AddFB2 / atomic commit
  -> plane / crtc / connector
  -> DE / TCON / HDMI / DSI
  -> 显示屏

重点：

PVR = render / memory / sync / dma-buf producer
sunxi-drm = KMS display owner / scanout consumer

PVR 自己的 drm_device 主要是 render + bridge 外壳，不是实际显示控制器 owner。真正拥有 KMS 显示资源的是 sunxi-drm。

 fbdev / Display Class 旧路线

fbdev 路线需要启用：

SUPPORT_DISPLAY_CLASS := 1

这样 pvrsrvkm 才会编进：

dc_server.o
server_dc_bridge.o

其链路是：

PVR userspace
  -> DRM_IOCTL_PVR_SRVKM_CMD
  -> PVR bridge
  -> DC bridge
  -> dc_server
  -> dc_sunxi
  -> registered_fb[]
  -> fb_set_var / fb_pan_display
  -> sunxi disp2
  -> 显示屏

重点：

PVR = render + Display Class present 调度
dc_sunxi = PVR Display Class 到 Linux fbdev 的适配层
sunxi disp2 fbdev = 真正驱动显示硬件

这条路线不走 KMS ioctl，不用 AddFB2 / atomic commit，而是最终通过 fbdev 的 fb_pan_display() / fb_set_var() 翻屏。

 PVR bridge 是什么

PVR bridge 不是 DRM 显示 bridge。

它是：

PVR userspace -> PVR kernel services

之间的一套私有 ioctl 分发机制。

入口是：

DRM_IOCTL_PVR_SRVKM_CMD

用户态传入：

bridge_id
bridge_func_id
in_data
out_data

内核通过：

PVRSRV_BridgeDispatchKM / BridgedDispatchKM

分发到具体功能，比如：

render bridge
memory bridge
sync bridge
dma-buf bridge
Display Class DC bridge

所以“渲染命令通过 PVR bridge 进入 RGX”的意思是：用户态把请求打包成 PVR 私有 ioctl，内核 bridge dispatcher 再分发到 RGX kernel services。

两条路线的本质区别

可以这样概括：

共用部分：
  PVR Services 核心
  GPU render
  memory
  sync
  command submit
  PVR private bridge

不同部分：
  显示 / present / scanout backend

DRM/KMS：

PVR 渲染 -> dma-buf -> sunxi-drm KMS -> 显示屏

fbdev / Display Class：

PVR 渲染 -> PVR DC bridge -> dc_sunxi -> fbdev -> sunxi disp2 -> 显示屏

所以不是“渲染阶段换不同回调”，而是 显示提交 / present 阶段的 backend 不同。

 fbdev + GPU 是否要求 GPU 驱动承担更多显示职责？

是的，但要准确表述：

fbdev + GPU：
  PVR 不只是负责渲染，
  还要启用 Display Class，
  通过 dc_sunxi 参与 present / 送显流程。

DRM/KMS + GPU：
  PVR 基本只负责渲染、buffer、sync、dma-buf；
  显示由 sunxi-drm 负责。

但这不代表 GPU 硬件直接控制 HDMI / DSI / TCON。真正操作显示硬件的仍然是 fbdev/disp2 或 sunxi-drm。

OpenGL ES / EGL 用户态库是否需要匹配后端？

需要。

不是说整个 libGLESv2.so 都要重写，而是：

GLES 渲染核心：
  大体仍走 PVR render bridge

EGL / window system / swap/present：
  必须和 DRM/KMS 或 fbdev/DC 后端匹配

差异主要发生在：

eglGetDisplay
eglCreateWindowSurface
eglSwapBuffers
buffer allocation
dma-buf export/import
Display Class DC configure
present / page flip

所以：

DRM/KMS 路线：
  需要支持 KMS / dma-buf / PRIME / native fence

fbdev/DC 路线：
  需要支持 Display Class / dc_sunxi / fbdev present

如果用户态库和内核后端不匹配，可能出现 eglInitialize、eglCreateWindowSurface、eglSwapBuffers 或 present 失败。
