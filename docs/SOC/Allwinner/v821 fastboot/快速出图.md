## 1. 总体结论

`perf2_fastboot` 的快速出图核心是：

1. **BOOT0/固件阶段提前启动 E907 FreeRTOS 镜像**  
   `sys_partition.fex` 里有独立 `riscv0` 分区，烧录 `amp_rv0.fex`，即 E907 侧固件。

2. **E907 先于 Linux 完成 Camera 链路初始化**  
   E907 的 `cpu0_app_entry()` 启动 OpenAMP 线程，同时主线程直接执行 `csi_init()`，把 sensor、TWI、MIPI/CSI、VIN、ISP、Scaler 等硬件链路先拉起来。

3. **E907 初始化完成后通过 `rpmsg_notify()` 告诉 Linux 哪些设备可以继续 probe**  
   通知名包括 `twi0`、`isp0`、`scaler0`、`scaler4`、`vinc0`、`vinc4` 等。

4. **Linux 内核 fastboot 配置把相关驱动提前到更早 initcall，并使用 delay/deferred 机制减少阻塞**  
   例如 `CONFIG_AW_IPC_FASTBOOT=y`、`CONFIG_VIN_INIT_MELIS=y`、`CONFIG_VIDEO_RT_MEDIA=y`、`CONFIG_AW_RPROC_FAST_BOOT=y` 等。

5. **initramfs 的 `/rdinit` 在正式 rootfs 挂载前立即执行 `demo_video_in`**  
   也就是 Linux 刚挂载 `/proc`、`/sys`、`/tmp`、`/dev` 后，第一条实际业务命令就是采集/编码视频。

6. **SPI Flash 控制权延后交给 Linux**  
   rdinit 里先运行 `demo_video_in`，然后才通过 `/proc/sys/kernel/deferred_device` 释放 `spif` 设备，挂载正式 squashfs rootfs。

所以这套方案的关键不是“Linux 启动很快以后再初始化 camera”，而是 **Camera 已经由 E907 提前热启动，Linux 到了 initramfs 阶段直接用现成链路采集第一帧/第一路码流**。

## 14. 为什么它比普通 Linux Camera 启动快

普通启动路径通常是：

```text
Linux kernel boot
  -> rootfs mount
  -> init/system service
  -> app start
  -> camera driver probe
  -> sensor power on
  -> sensor i2c init
  -> mipi/csi/vin/isp init
  -> AE/AWB 收敛
  -> app request buffer
  -> streamon
  -> first frame
```

`perf2_fastboot` 路径变成：

```text
E907 与 Linux 并行启动
  -> E907 提前完成 sensor/vin/isp 预热和 AE
  -> Linux 早期注册 rt-media
  -> initramfs 第一条命令直接 demo_video_in
  -> 从已经 ready 的链路拿帧
```

优化点包括：

1. **双核并行**：Camera 初始化不占用 Linux 启动临界路径。
2. **远端核提前预热 sensor/ISP**：I2C、上电、MIPI、前几帧、AE 收敛都提前做。
3. **delay_init + rpmsg_notify**：Linux 不盲目等待硬件初始化，而是等远端 ready 通知。
4. **initramfs 直接出图**：绕过完整 rootfs 和复杂 init 服务。
5. **SPIF 延迟初始化**：先出图，再挂载正式 rootfs，减少 IO 阻塞。
6. **RT Media 提前 initcall**：`/dev/rt-media` 更早可用。
7. **预加载最小依赖**：`demo_video_in`、`librt_media.so` 等被放进预加载集合。

## 15. 关键配置/源码检查清单

如果调试 fastboot 快速出图，建议按下面顺序查。

### E907 侧

1. `amp_rv0.fex` 是否正确打包到 `riscv0` 分区。
2. `CONFIG_PROJECT_V821_E907_PERF2_FASTBOOT=y`。
3. `CONFIG_COMPONENTS_OPENAMP=y`。
4. `CONFIG_COMPONENTS_RPBUF=y`。
5. `CONFIG_RPMSG_NOTIFY=y`。
6. `CONFIG_SLAVE_EARLY_BOOT=y`。
7. `csi_init()` 是否成功打印 `csi init success!`。
8. 是否发出 `rpmsg_notify("vinc0")`、`rpmsg_notify("isp0")` 等。

### Linux kernel 侧

1. `CONFIG_AW_IPC_FASTBOOT=y`。
2. `CONFIG_AW_RPROC_FAST_BOOT=y`。
3. `CONFIG_AW_REMOTEPROC_E907_BOOT=y`。
4. `CONFIG_AW_RPMSG_NOTIFY=y`。
5. `CONFIG_AW_RPBUF_CONTROLLER_SUNXI=y`。
6. `CONFIG_VIN_INIT_MELIS=y`。
7. `CONFIG_ISP_SERVER_MELIS=y`。
8. `CONFIG_VIDEO_RT_MEDIA=y`。
9. `/dev/rt-media` 是否在 `/rdinit` 前可用。
10. VIN/ISP/scaler 节点的 `delay_init = <1>` 和 `rpmsg-ser-name` 是否匹配。

### initramfs 侧

1. `/rdinit` 是否确实被执行。
2. `demo_video_in` 和相关 so 是否在 initramfs 里。
3. `mount -t devtmpfs none /dev` 后是否能看到 `/dev/rt-media`。
4. `demo_video_in -n 40 -s0 1280x720 -f0 0` 是否返回错误。
5. SPIF deferred 是否放在 `demo_video_in` 后面。

### 设备树/内存

1. `e907_mem_fw` 地址和 E907 固件链接地址是否一致。
2. `rpbuf_controller0` 的 memory-region 是否指向 `e907_rpbuf_reserved`。
3. `isp00` 是否配置 `rpbuf = <&rpbuf_controller0>`。
4. `isp-region = <&isp_dram_reserved>` 是否存在。
5. `size_pool`、`linux,cma` 是否足够。

## 16. 一句话概括

V821 `perf2_fastboot` 快速出图的本质是：

**把 Camera 冷启动中最耗时的 sensor 上电、I2C 配置、VIN/ISP 初始化、前几帧等待和 AE 收敛前移到 E907 FreeRTOS，并与 Linux 启动并行；Linux 通过 OpenAMP/RPMsg/RPBuf 接管已预热的硬件链路，在 initramfs 的 `/rdinit` 阶段直接运行 `demo_video_in`，从而在正式 rootfs 启动前就拿到图像/码流。**
