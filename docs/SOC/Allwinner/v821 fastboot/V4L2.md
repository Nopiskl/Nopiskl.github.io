## 1. 总体结论

这个 SDK 里已经有一套“Linux 上层 V4L2 保持不变、底层部分控制权交给 RTOS”的框架，核心是：

- **Linux app 仍打开 `/dev/videoX`，调用 `VIDIOC_S_FMT / REQBUFS / QBUF / STREAMON / DQBUF / STREAMOFF`。**
- Linux VIN 驱动仍负责：
  - V4L2 设备节点；
  - vb2 buffer 队列；
  - frame buffer 管理；
  - DMA buffer 出帧；
  - 用户态 ioctl 入口；
  - 部分 IRQ/DMA 回调。
- RTOS/E907 负责：
  - sensor 上电、I2C/TWI、格式切换、stream 开关；
  - ISP server；
  - 3A；
  - ISP 参数加载/保存；
  - 部分 MIPI/CSI/ISP/scaler/VINC 控制；
  - 在 fastboot 阶段先启动 camera pipeline。
- Linux 与 RTOS 之间通过两类机制通信：
  1. **rpmsg 小消息**：控制命令、状态切换、start/stop ACK、ISP 控制命令。
  2. **rpbuf 共享内存**：大块 ISP 参数、统计信息、load/save buffer。

所以要实现“Linux app 操作 V4L2 时转发 rpmsg 到 RTOS 控硬件”，不是让用户态 app 直接用 rpmsg，而是：

> 在 Linux VIN/V4L2 驱动内部拦截 V4L2 操作，把真正需要硬件控制的动作转成 `isp_rpmsg_send()` 或 `vinc_status_rpmsg_send()`，RTOS 侧收到后调用自己的 VIN/sensor/ISP 函数；Linux 侧仍保留 V4L2 ABI 和 buffer/DMA 数据路径。

## 7. 推荐实现方案：保持 V4L2 ABI，内核驱动转发硬件控制

### 7.1 分层思路

建议把操作分成三类：

#### A. Linux 必须保留本地处理的操作

这些不要转给 RTOS，或者只把结果同步给 RTOS：

- `VIDIOC_QUERYCAP`
- `VIDIOC_ENUM_FMT`
- `VIDIOC_G_FMT`
- `VIDIOC_REQBUFS`
- `VIDIOC_QUERYBUF`
- `VIDIOC_QBUF`
- `VIDIOC_DQBUF`
- `mmap`
- `poll`
- vb2 buffer state
- DMA buffer 地址管理

原因：这些属于 Linux V4L2 ABI 和用户态 buffer 流转。转给 RTOS 没意义，反而会破坏用户态兼容性。

#### B. 需要转发给 RTOS 的硬件控制操作

建议转发：

- `VIDIOC_S_INPUT`
- `VIDIOC_S_FMT`
- `VIDIOC_S_PARM`
- `VIDIOC_STREAMON`
- `VIDIOC_STREAMOFF`
- exposure/gain/AE/AWB/IR 等 control；
- 私有 ISP attr ioctl；
- sensor mode switch；
- mipi switch；
- reset/start/stop ISP server；
- close ISP。

这些操作最终会影响 sensor、MIPI、CSI、ISP、scaler、VINC 寄存器，适合由 RTOS 统一控制。

#### C. 双方都需要知道的状态

需要同步：

- 当前 sensor id；
- 分辨率、pixelformat、fps；
- wdr mode；
- IR 状态；
- streaming 状态；
- buffer 物理地址或 DMA queue 状态；
- ISP server 是否 ready；
- 控制权是 Linux 还是 RTOS。

## 8. 具体转发链路设计

### 8.1 Linux app 调用流程

用户态不变：

内核内部变成：

```text
vin_open()
  -> rpmsg: ARM_VIN_START
  -> wait RTOS ACK 或检查 control = CONTROL_BY_ARM

vidioc_s_input()
  -> 保存 Linux 本地 pipe/sensor_sel
  -> rpmsg: VIN_SET_V4L2_IOCTL / VIN_SET_SENSOR_INFO

vidioc_s_fmt()
  -> 更新 Linux frame fmt
  -> rpmsg: VIN_SET_V4L2_IOCTL(S_FMT, width, height, pixfmt, field, fps...)

REQBUFS/QBUF
  -> Linux 本地 vb2 处理，不转 RTOS
  -> 如果 RTOS 需要 DMA 地址，再额外同步 buffer info

vidioc_streamon()
  -> vb2_ioctl_streamon()
  -> rpmsg: VIN_SET_ISP_START 或 VIN_SET_V4L2_IOCTL(STREAMON)
  -> RTOS 执行 vin_s_stream(id, PWR_ON)
  -> Linux 设置 VIN_STREAM，启动 timer/irq/dma 相关状态
  -> DQBUF 从 Linux DMA/VIN ISR 返回 frame

vidioc_streamoff()
  -> rpmsg: VIN_SET_CLOSE_ISP / VIN_SET_V4L2_IOCTL(STREAMOFF)
  -> RTOS 执行 vin_s_stream(id, PWR_OFF)
  -> Linux vb2_ioctl_streamoff()
```

## 9. STREAMON 应该怎么改

### 9.1 不建议完全跳过 Linux pipeline

Linux 仍要能 DQBUF，说明 Linux 侧 DMA/IRQ/buffer 状态必须正常。不能简单地“STREAMON 只发 rpmsg，Linux 什么都不做”。

否则会出现：

- app `DQBUF` 卡住；
- buffer 不 complete；
- poll 无事件；
- Linux 不知道 frame done；
- IRQ 被 RTOS 占着，Linux 收不到；
- 或 RTOS 正在写 DMA，Linux buffer 队列不知道。

### 9.2 正确做法

`VIDIOC_STREAMON` 里：

1. Linux 先完成 vb2 队列进入 streaming；
2. 根据当前控制模式判断：
   - 如果 RTOS 已经初始化 sensor/ISP，只需要 RTOS start 或 return control；
   - 如果 Linux 要接管 DMA IRQ，则 RTOS 发 `"return"` 给 `vincX`，Linux request_irq；
   - 如果 RTOS 仍控制硬件，则 Linux 只作为 buffer consumer，需要额外共享 buffer queue 给 RTOS。
3. 发 rpmsg 给 RTOS：
   - `VIN_SET_V4L2_IOCTL: STREAMON`
   - 或 `VIN_SET_ISP_START`
4. 等 ACK，成功后 `set_bit(VIN_STREAM)`。
5. 再进入正常 DQBUF。

简化伪代码：

```c
static int vidioc_streamon(...)
{
	ret = vb2_ioctl_streamon(file, priv, type);
	if (ret)
		return ret;

	if (use_rtos_hw_control) {
		ret = vin_send_v4l2_ioctl_to_rtos(vinc, VIN_OP_STREAMON, ...);
		if (ret)
			goto err_vb2_streamoff;

		ret = vin_wait_rtos_ack(vinc, seq, timeout);
		if (ret)
			goto err_vb2_streamoff;

		set_bit(VIN_STREAM, &cap->state);
		return 0;
	}

	ret = vin_pipeline_call(vinc, set_stream, &cap->pipe, vinc->stream_idx);
	...
}
```

## 10. S_FMT / S_INPUT / S_PARM 怎么转

### 10.1 `VIDIOC_S_FMT`

Linux 本地仍要设置 `cap->frame`，因为 buffer size、plane、bytesperline 都依赖它。然后把最终格式发给 RTOS：

```text
Linux vidioc_s_fmt:
  - try/adjust format
  - update cap->frame
  - calculate buffer size
  - rpmsg VIN_OP_S_FMT(width/height/pixfmt/fps)
RTOS:
  - 找到 video_id 对应 global_video[id]
  - 设置 o_width/o_height/fps/wdr
  - sensor_g_format 或 sensor_g_switch_format
  - sunxi_isp_s_mbus_config / mipi / csi config
  - ACK
```

### 10.2 `VIDIOC_S_INPUT`

Linux 选择 sensor/input 后同步 RTOS：

```text
VIN_OP_S_INPUT:
  video_id
  sensor_id
  mipi_sel
  isp_sel
  csi_sel
```

RTOS 收到后更新 `global_video[id].rear_sensor`、`mipi_sel`、`isp_sel` 等。

### 10.3 `VIDIOC_S_PARM`

主要同步 fps：

```text
VIN_OP_S_PARM:
  fps_num
  fps_den
```

RTOS 侧更新：

```c
isp_get_cfg[isp_id].sensor_get_fps = fps;
```

再由 sensor driver 或 ISP server 生效。

## 11. control/ioctl 怎么转发

### 11.1 ISP attr 已经有现成模式

当前 `vin_set_isp_attr_cfg_ctrl()` 已经把很多 ISP 控制通过 rpmsg 发给 RTOS，例如：

- 降噪强度；
- 3DNR；
- PLTM/WDR；
- IR；
- EV；
- AE lock；
- AE table；
- AE ROI；
- MIPI switch。

这部分无需重做，只要确认 RTOS 侧 ISP server 实现了对应 `VIN_SET_ATTR_IOCTL` 处理。

### 11.2 sensor 控制建议也走统一 V4L2 ctrl 转发

例如 Linux app 用标准 V4L2 ctrl：

- `V4L2_CID_EXPOSURE`
- `V4L2_CID_GAIN`
- `V4L2_CID_AUTO_WHITE_BALANCE`
- `V4L2_CID_HFLIP`
- `V4L2_CID_VFLIP`

当前 `vin_s_ctrl()` 里很多 ctrl 最终会 `v4l2_s_ctrl(NULL, sensor->ctrl_handler, &c)`，即 Linux sensor subdev 控。若硬件由 RTOS 控制，需要改成：

```text
vin_s_ctrl()
  if use_rtos_hw_control:
      rpmsg VIN_OP_S_CTRL(ctrl_id, value)
      wait ack
  else:
      v4l2_s_ctrl(sensor->ctrl_handler)
```

RTOS 侧：

```text
VIN_OP_S_CTRL:
  switch ctrl_id:
    EXPOSURE -> sensor_core->s_exp_gain()
    GAIN -> sensor_core->s_exp_gain()
    IR -> sensor_core->s_ir_status()
    HFLIP/VFLIP -> sensor-specific ioctl or register write
```

## 12. 最关键的问题：谁控制 DMA/IRQ？

实现能否成功，核心不是 rpmsg，而是 **Linux 和 RTOS 不能同时控制同一套 VIN DMA/IRQ/寄存器**。

现有 notify 机制已经考虑这个问题：

- RTOS 发 `"get"`：Linux 释放 IRQ；
- RTOS 发 `"return"`：Linux 重新申请 IRQ；
- Linux 发 `ARM_VIN_START/STOP`：告诉 RTOS Linux 要开始/停止；
- RTOS 发 ACK 或状态变更。

这意味着有两种架构路线。

### 路线 A：RTOS 只控制 sensor/ISP，Linux 控制 VIN DMA 出帧

这是最适合“Linux 上层仍能使用 V4L2”的方案。

- RTOS 控：
  - sensor power；
  - sensor stream；
  - exposure/gain；
  - ISP server；
  - IR；
  - mipi/sensor mode。
- Linux 控：
  - V4L2 node；
  - vb2 queue；
  - VIN DMA；
  - IRQ；
  - DQBUF。

优点：

- Linux app 兼容性最好；
- 不需要 RTOS 管 Linux buffer queue；
- DQBUF 正常；
- 只需同步硬件配置和 sensor/ISP 控制。

缺点：

- MIPI/CSI/ISP/VIN pipeline 里有些寄存器控制边界要拆清楚；
- RTOS 和 Linux 不能重复初始化同一模块。

建议你优先走这个方案。

### 路线 B：RTOS 控整条 camera pipeline，Linux 只消费共享帧

这种方案更像“RTOS camera server + Linux V4L2 wrapper”。

- RTOS 控：
  - sensor；
  - MIPI；
  - CSI；
  - ISP；
  - scaler；
  - VINC DMA；
  - frame buffer。
- Linux 控：
  - `/dev/videoX`；
  - 把 RTOS 提供的帧包装成 V4L2 buffer。

这会复杂很多，因为必须实现：

- buffer 共享；
- RTOS 写入 Linux QBUF 的物理地址；
- cache sync；
- frame done 通知；
- Linux vb2 buffer complete；
- 掉帧/超时处理；
- 多 buffer ownership。

如果 SDK 没有现成 rpbuf frame queue，不建议一上来做这个。

## 13. 推荐落地方案

### 阶段 1：复用现有控制权机制

打开：

- `CONFIG_RV_RUN_CAR_REVERSE`
- `CONFIG_ISP_SERVER_MELIS`
- `CONFIG_VIN_INIT_MELIS`
- 相关 rpmsg/rpbuf/openamp 配置。

确认启动日志里有：

```text
isp0 rpmsg probe
rpmsg_notify ready
ARM_RPMSG_READY
sunxi,notify
```

Linux open 时允许：

```text
vin_open()
  -> ARM_VIN_START
RTOS:
  -> GET_BY_LINUX
  -> arm_start
Linux:
  -> CONTROL_BY_ARM
```

### 阶段 2：把 V4L2 S_FMT/S_PARM/S_CTRL 转成 rpmsg

优先改这些：

1. `vin_s_ctrl()`
2. `vin_set_isp_attr_cfg_ctrl()`
3. `vidioc_s_parm()`
4. `vidioc_s_fmt_vid_cap_mplane()`

不要先动 buffer 和 DMA。

### 阶段 3：STREAMON/STREAMOFF 只让 RTOS 做 sensor/ISP start/stop

在 `vidioc_streamon()`：

- Linux 仍调用 vb2；
- Linux 仍保留 IRQ；
- 发 rpmsg 给 RTOS：
  - sensor stream on；
  - ISP server start；
  - sensor exp/gain 初始化；
- RTOS 不要抢 Linux VIN DMA IRQ。

在 `vidioc_streamoff()`：

- 先通知 RTOS sensor/ISP stop；
- Linux 再停本地 DMA/vb2；
- 或反过来，根据硬件时序决定，但必须保证不会在 Linux DMA 运行时关闭 sensor 导致异常中断风暴。

### 阶段 4：如果确实要 RTOS 控 VINC DMA，再设计 frame rpbuf

只有在你明确要 RTOS 控整条 pipeline 时，再做：

- Linux QBUF 后把 buffer 物理地址通过 rpmsg/rpbuf 发给 RTOS；
- RTOS DMA 到该地址；
- RTOS frame done rpmsg 回 Linux；
- Linux `vb2_buffer_done()`。

这相当于重写一个 V4L2 backend，工作量明显更大。

## 16. 最终建议

如果你的目标是“Linux app 通过 V4L2 使用，硬件控制放 RTOS”，建议采用：

```text
Linux V4L2 app
    ↓
Linux vin_video.c
    - 保留 V4L2/vb2/buffer/DMA
    - 拦截 S_FMT/S_PARM/S_CTRL/STREAMON/STREAMOFF
    ↓
Linux vin_rp.c / vin_core.c
    - isp_rpmsg_send()
    - vinc_status_rpmsg_send()
    ↓
rpmsg/openamp
    ↓
RTOS vin_rpmsg.c + ISP server handler
    - sensor_power()
    - sensor_g_format()
    - sensor_core->s_stream()
    - sensor_core->s_exp_gain()
    - sunxi_isp_update/reset/start
    ↓
硬件 sensor / ISP / MIPI / CSI
```

关键原则：

1. **用户态 ABI 不变。**
2. **Linux 继续管理 V4L2 buffer 和 DQBUF。**
3. **RTOS 控 sensor/ISP 实时硬件。**
4. **不要 Linux 和 RTOS 同时写同一个 I2C/ISP/VIN 寄存器。**
5. **STREAMON/STREAMOFF 必须有 ACK 和超时恢复。**
6. **大数据走 rpbuf，小命令走 rpmsg。**
7. **先做 sensor/ISP 控制转发，不要一开始就让 RTOS 接管 VINC DMA。**

这样改动最小，也最符合当前 SDK 已经存在的 `CONFIG_ISP_SERVER_MELIS`、`vin_rpmsg`、`isp_rpmsg_send()`、`vinc_status_rpmsg_send()` 这套设计。
