## 第 3 轮

### 用户

那结合uboot源码分析一下，既然没有 sc3917 专属的 U-Boot DTS，那它uboot初始化驱动时用的dts是哪里来的？


### Codex

它初始化驱动时用的不是 sc3917 专属的 U-Boot DTS，而是运行时传进 U-Boot 的 DTB/FDT blob。这份 blob 来自打包阶段生成的 sunxi.fex，也就是 boot_package 里的 dtb item。

源码链路是这样的：

当前 U-Boot 配置是 CONFIG_OF_BOARD=y，CONFIG_OF_SEPARATE 没开：

sun50iw10p1_tina_defconfig (line 8)

U-Boot 启动早期会先调用 fdtdec_setup()，再初始化 DM 驱动模型：

common/board_f.c (line 800)

其中 fdtdec_setup() 会把 gd->fdt_blob 设置好。

在 CONFIG_OF_BOARD 模式下，fdtdec_setup() 调用 board_fdt_blob_setup()：

lib/fdtdec.c (line 1453)

Allwinner 的实现是：先检查 U-Boot 镜像尾部 _end 后面有没有 DTB；如果没有，就去 CONFIG_SYS_TEXT_BASE + SUNXI_DTB_OFFSET 找：

arch/arm/mach-sunxi/board.c (line 149)

当前配置里：

CONFIG_SYS_TEXT_BASE=0x4A000000
SUNXI_DTB_OFFSET=(2<<20)
所以 U-Boot 预期 DTB 地址大约是：

0x4A000000 + 0x200000 = 0x4A200000
这份 DTB 是打包脚本生成并放进 boot package 的。pack_img.sh 会生成 sunxi.dtb，再复制成 sunxi.fex：

pack_img.sh (line 1140)


pack_img.sh (line 1252)

然后 boot_package.cfg 明确把它作为 dtb item 打包进去：

boot_package.cfg (line 9)

所以一句话总结：

U-Boot 驱动初始化用的 FDT 是 boot0/SPL 从 boot_package.fex 里的 dtb item，也就是 sunxi.fex，加载到 0x4A200000 附近后交给 U-Boot 的。这个 sunxi.fex 来自打包阶段用 kernel/device 侧 DTS 生成的 DTB，不是来自 u-boot-2018/arch/arm/dts/ 下的 sc3917 专属 DTS。

因此 device/config/chips/r818/configs/sc3917/board.dts 对它是有影响的，只是路径不是“U-Boot 自己编译一个 sc3917 U-Boot DTS”，而是：

```text
sc3917/board.dts + sys_config.fex
    -> pack_img.sh 生成 sunxi.dtb
    -> sunxi.dtb 复制为 sunxi.fex
    -> boot_package.cfg 打包为 dtb item
    -> boot0 加载到 U-Boot 预期地址
    -> U-Boot 的 gd->fdt_blob 指向它
    -> DM/驱动从 gd->fdt_blob 扫描节点
```

## 第 4 轮

### 用户

请你结合这份官方PDF DshanPI-ROSx-TinaSDK\DshanPI-ROSx-TinaSDK-V1\docs\Linux_U-Boot_开发指南.pdf
分析一下：既然这样，是不是uboot和kernel用同一份dtb？一开始先是uboot使用，修改使用完以后传给kernel接着用？


### Codex

对当前 r818-sc3917 / linux-4.9 / u-boot-2018 这套配置来说：基本可以理解为 U-Boot 和 Kernel 使用同一份 DTB 流水线产物。

但要加两个限定：

不是 U-Boot 自己编译出来的独立 u-boot.dtb
当前 sun50iw10p1_tina_defconfig 是 CONFIG_OF_BOARD=y，CONFIG_OF_SEPARATE 没开，所以 U-Boot 启动时拿的是外部传入的 FDT，而不是 u-boot-2018/arch/arm/dts 编进来的 DTB。

Kernel 拿到的是 U-Boot 修改后的那份 working FDT
U-Boot 先把 boot package 里的 dtb item，也就是 sunxi.fex/sunxi.dtb，作为 gd->fdt_blob 使用。之后会修改这份 FDT，例如启用对应存储节点、更新 dram 参数、charger_mode、display、MAC、PMU 等。最后 boot kernel 时把当前 gd->fdt_blob 作为 images.ft_addr 传给 Linux。

源码链路很明确：

U-Boot 获取 FDT：

arch/arm/mach-sunxi/board.c (line 149)

没有镜像尾部 DTB 时，去 CONFIG_SYS_TEXT_BASE + SUNXI_DTB_OFFSET 找。

当前配置固定地址大约是：
0x4A000000 + 0x200000 = 0x4A200000

boot package 里有 dtb：

boot_package.cfg (line 9)

U-Boot 修改传给 kernel 的 FDT：

board_common.c (line 705)


board_helper.c (line 665)

boot kernel 时直接用当前 gd->fdt_blob：

common/bootm.c (line 307)

ARM64 跳转 Linux 时把 images->ft_addr 传进去：

arch/arm/lib/bootm.c (line 409)

结合官方 
Linux_U-Boot_开发指南.pdf
 第 5.3 和 5.3.3.3 的描述，它区分了 “U-Boot dts / kernel dts” 以及运行时 “内部 dts 阶段 / 内核 dts 阶段”。但你这个 r818-sc3917 当前并没有走“U-Boot 内部独立 dts 再切到 kernel dts”的模式，而是一开始就用外部/kernel 侧 DTB 作为 U-Boot working FDT。

所以最终答案是：

```text
sc3917 board.dts / sys_config.fex
    -> pack 阶段生成 sunxi.dtb
    -> 复制为 sunxi.fex
    -> 打进 boot_package.fex 的 dtb item
    -> boot0 加载到 U-Boot 预期地址
    -> U-Boot 作为 gd->fdt_blob 使用并修改
    -> boot Linux 时把修改后的 gd->fdt_blob 传给 kernel
```

也就是：当前这套配置下，U-Boot 和 Kernel 用的是同一份 DTB 的运行时实例，U-Boot 会先用并修改它，然后 Kernel 接着用修改后的版本。

## 第 5 轮

### 用户

那请你再根据更新的SDK，结合你之前的总结，分析一下，a733的boot packae是TINA-AIOT\tina-aiot\device\config\chips\a733\configs\default\boot_package.cfg，它却没有打包sunxi.fex，uboot仍然可以加载dts，内核也能加载dtb，为什么？

### Codex

结论：a733 不是 r818 那种 boot_package.cfg 里单独打包 sunxi.fex 给 U-Boot 用的流程。a733 是 U-Boot 自带一份 DTB，Kernel 的 DTB 另外打进 boot.img/boot.fex，启动后 U-Boot 再把工作 FDT 切到 Kernel DTB。

所以 boot_package.cfg 里没有 sunxi.fex 仍然正常。

1. U-Boot 为什么能用 DTS？

a733 的 U-Boot defconfig 指定了默认设备树：

也就是说，a733 的 U-Boot 使用 OF_SEPARATE：U-Boot 编译时会生成 u-boot.dtb / dts/dt.dtb，并把它和 U-Boot 镜像组合进 u-boot.fex。所以 boot_package.cfg 只需要：

```text

item=monitor, monitor.fex
item=scp,     scp.fex
```

见 
boot_package.cfg (line 3)
。

U-Boot 的 DTS 主文件是：


sun60iw2p1-soc-system.dts (line 16)
它里面还会 include 板级 U-Boot DTS：

这个 .board-uboot.dts 是 U-Boot Makefile 构建时生成的：优先从当前板级目录的 uboot-board.dts 复制，没有则退回 common board DTS。逻辑在

2. Kernel 为什么也能拿到 DTB？

Kernel 的 DTB 不是通过 boot_package.cfg 里的 sunxi.fex 传的，而是构建 boot.img 时打进去的。

mkkernel.sh 里可以看到 DTB 来源是：

local DTB="$STAGING_DIR/sunxi.dtb"
后面 mkbootimg 会带上类似：

--dtb ${DTB}
也就是把 sunxi.dtb 打进 boot.img。而分区表里 boot 分区下载的是 boot.fex，本质就是打包后的 boot image。

3. 启动时 DTS/DTB 的切换流程

a733 defconfig 打开了：


sun60iw2p1_a733_defconfig (line 24)

CONFIG_SUNXI_NECESSARY_REPLACE_FDT=y
所以 U-Boot 启动时流程大概是：

早期 U-Boot 先用自己内置/附带的 U-Boot DTB，初始化 pinctrl、clock、mmc、flash、DM 驱动等。
到 board_late_init() 阶段，执行 sunxi_replace_fdt_v2()，见 
board_common.c (line 823)
。
sunxi_replace_fdt_v2() 会优先找 dtb 分区；如果没有，就从 boot 分区里的 Android boot image 中解析 DTB，见 
sunxi_replace_fdt.c (line 274)
。
解析 boot image DTB 的逻辑在 
image-android.c (line 581)
。
找到 Kernel DTB 后，U-Boot 把 gd->fdt_blob / working FDT 切换到这份外部 DTB，再修补启动参数，然后传给 Kernel。
所以这里不是“从头到尾共用同一份 DTB”，而是：

U-Boot 早期：使用 U-Boot 自己的 DTB，随 u-boot.fex 内置/附带
Kernel 启动前：U-Boot 从 boot.img/boot.fex 取出 Kernel DTB
Kernel 启动时：使用被 U-Boot 修补后的 Kernel DTB
4. 和 r818 的关键区别

r818 老流程更像：

boot_package.fex 里单独有 item=dtb,sunxi.fex
U-Boot 直接拿这份外部 DTB 用
后面再传给 Kernel
a733 新流程则是：

boot_package.fex 里没有 sunxi.fex
U-Boot DTB 在 u-boot.fex 里面
Kernel DTB 在 boot.img/boot.fex 里面
U-Boot 后期 replace fdt，把工作 DTB 换成 Kernel DTB
因此，a733 没有打包 sunxi.fex，并不代表没有 DTB；只是 U-Boot DTB 和 Kernel DTB 的存放位置变了。
