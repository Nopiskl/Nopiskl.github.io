## VIN 框架中 CSI 节点的本质

**CSI 节点（`csi@N`）在全志 VIN 框架里就是 CSIC PRS（CSI Controller Parser），即"图像数据解析器/接收器"**，不是 MIPI CSI 协议本身。

### 各节点职责分工

| 节点 | 硬件模块 | 职责 |
|---|---|---|
| `mipi@N` | MIPI D-PHY + combo PHY | 负责物理层：接收差分信号、lane 解串、时钟恢复，管理 `mipib_pins_a` 等引脚 |
| `csi@N` | CSIC PRS（Parser） | 负责协议层：解析 MIPI CSI-2 包头、提取像素数据、并行 NCSI 时序解析 |
| `isp@N` | ISP | 图像信号处理 |
| `vipp@N` / `scaler@N` | VIPP/Scaler | 缩放裁剪输出 |

## csi0、csi1、csi2 的区别

三者**驱动完全相同**（都是 `allwinner,sunxi-csi`），本质上都是 CSIC Parser，区别只有：

| 节点 | 寄存器基地址 | 中断号 | device_id | pinctrl |
|---|---|---|---|---|
| `csi0` | `0x05820000` | SPI 100 | 0 | **无**（纯 MIPI 接收，不需要并行引脚）|
| `csi1` | `0x05821000` | SPI 101 | 1 | **无**（同上）|
| `csi2` | `0x05822000` | SPI 102 | 2 | 原本有 `ncsi_pins_a`（PE0~PE15，并行 NCSI 接口）|

**核心区别：**

- **csi0 / csi1**：纯粹接收来自 mipi0/mipi1 解包后的数据，**不涉及物理引脚**
- **csi2**：除了可以接收 MIPI 数据外，还额外带了并行 NCSI 接口的 pinctrl（`ncsi_pins_a`，即 PE 组），用于连接并行摄像头

```1359:1370:lichee/linux-4.9/arch/arm/boot/dts/sun8iw21p1.dtsi
    compatible = "allwinner,sunxi-csi";
    reg = <0x0 0x05822000 0x0 0x1000>;
    interrupts = <GIC_SPI 102 IRQ_TYPE_LEVEL_HIGH>;
    //pinctrl-names = "default","sleep";
    //pinctrl-0 = <&ncsi_pins_a>;
    //pinctrl-1 = <&ncsi_pins_b>;
    device_id = <2>;
    status = "okay";

```

csi2 已经没有任何 pinctrl，等同于 csi1，**PE0 冲突不会再出现**。

---

## 注意

直接修改 dtsi 会影响所有使用 sun8iw21p1 的板子。如果只是客户自己用，问题不大。但**更规范的做法**是在 board.dts 里用 `/delete-property/` 覆盖，保持 dtsi 原样。客户如果能接受现状就没问题，重新编译烧录后 `probe of csi2 failed with error -22` 报错就会消失。

```plaintext
csi2:csi@2 {
    device_type = "csi2";
    compatible = "allwinner,sunxi-csi";
    reg = <0x0 0x05822000 0x0 0x1000>;
    interrupts = <GIC_SPI 102 IRQ_TYPE_LEVEL_HIGH>;
    //pinctrl-names = "default","sleep";
    //pinctrl-0 = <&ncsi_pins_a>;
    //pinctrl-1 = <&ncsi_pins_b>;
    device_id = <2>;
    status = "okay";
};
```
